In `src/audio/AnalysisTick.ts`, immediately after the disambiguation engine block (after the closing brace of the `else` fallback at line ~214, before the cross-correlation block at line ~217), add a second-pass role reclassification loop:

```typescript
  // Phase 12 gap closure: Re-classify roles using displayActivityScore.
  // The first pass (line 143) used pre-disambiguation activityScore. Now that
  // runDisambiguationEngine has written displayActivityScore, re-run classifyRole
  // so that canvas node visuals (size, color) reflect disambiguated scores.
  for (const instr of instrs) {
    if (instr.displayActivityScore !== instr.activityScore) {
      const newRole = classifyRole(instr.displayActivityScore, instr.role);
      if (newRole !== instr.role) {
        instr.role = newRole;
        instr.roleSinceSec = state.audioCtx?.currentTime ?? 0;
        onRoleChange?.(instr.instrument, newRole);
      }
    }
  }
```

Key implementation details:
- Guard `if (instr.displayActivityScore !== instr.activityScore)`: Only re-classify instruments whose scores actually changed from disambiguation. Instruments not touched by any disambiguator have `displayActivityScore === activityScore` (set by the engine's initialization step), so they skip the second pass entirely. This avoids redundant classifyRole calls.
- The `classifyRole` call uses `instr.role` as the current role (which was set by the first pass). This preserves hysteresis behavior — the second pass starts from the first pass's role, not from the previous tick's role.
- `onRoleChange` fires only if the disambiguated score actually changes the role. If the first pass already assigned the correct role (e.g., both activityScore and displayActivityScore are in the same threshold band), no callback fires.
- Zero allocations: no new arrays, objects, or closures. Just a for-of loop with two comparisons and a function call.
