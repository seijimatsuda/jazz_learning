The `globalAlpha` set by `CanvasRenderer` before the `drawNode` call persists through the `drawNode` execution as expected.

#### Connection 4: Phase 14 reactive lineup → Phase 12 DisambiguationState reinitialization

**Status: CONNECTED — with one architectural note**

Trace:
1. `VisualizerCanvas.tsx` line 28: `const lineup = useAppStore(s => s.lineup)` — reactive Zustand selector, triggers re-render when lineup changes
2. `VisualizerCanvas.tsx` line 120: `useEffect(() => { ... }, [audioStateRef, lineup])` — effect re-runs when lineup changes, destroying old renderer and creating a new one with the new lineup
3. `App.tsx` line 55: `initAnalysisState(lineup, ...)` — analysis state initialized with lineup at calibration time
4. `App.tsx` line 59: `audioStateRef.current.disambiguation = initDisambiguationState()` — `DisambiguationState` initialized once at calibration, not per lineup change

**Architectural note (not a bug, but a constraint):** `initDisambiguationState()` is called only once at calibration time in `App.tsx`, not when the lineup changes after calibration. If the user changes the lineup post-calibration (e.g., switching from a quartet to a sextet with trombone added), the `CanvasRenderer` is re-created correctly (Phase 14's reactive lineup), but the `DisambiguationState` ring buffers are not re-initialized for the new instrument set. The `DisambiguationEngine` is guarded by `hasInstrumentPair` checks, so a newly added pair starts fresh from live audio data — the old ring buffer samples from the previous lineup's instruments don't cause incorrect output, but they do mean the tremolo RMS buffer may contain stale data for the first 2 seconds. This is an acceptable pre-existing tradeoff (not introduced by v1.2), not a wiring break.

### E2E Flow Verification

#### Flow 1: Audio load → overlapping instrument lineup → disambiguation → differentiated node visuals → confidence indicator

**Status: COMPLETE**

Steps verified:
1. User uploads audio → `isFileLoaded` becomes true → `App.tsx` calibration effect runs
2. `initAnalysisState(lineup, fftSize)` creates `InstrumentAnalysis` objects with `rawActivityScore` and `displayActivityScore` fields
3. `initDisambiguationState()` allocates `DisambiguationState` ring buffers
4. Every 100ms, `runAnalysisTick` fires:
   - Step 1: `computeActivityScore` → written to both `activityScore` and `rawActivityScore`
   - Step 2: First-pass `classifyRole(activityScore, role)`
   - Step 3: `runDisambiguationEngine` writes `displayActivityScore` (weighted by disambiguators)
   - Step 4: Second-pass `classifyRole(displayActivityScore, role)` — fires `onRoleChange` if role changed
5. `onRoleChange` → `useAppStore.getState().setInstrumentRole` → Zustand state updated
6. `onDisambiguationUpdate` → `disambiguationConfidence` cached in `CanvasRenderer.disambiguationConfidence`
7. Canvas render loop: for each instrument, `getInstrumentPairKey(instrument, instrumentOrder)` → `disambiguationConfidence[pairKey]` → `ctx.globalAlpha = 0.5` when confidence < 0.5
8. `drawNode` renders fill circle + ring stroke with dimmed alpha for uncertain instruments

For sax + keyboard lineup specifically: `SaxKeyboardDisambiguator` runs when `hasInstrumentPair(instrs, 'saxophone', 'keyboard') && chroma !== null`. Chroma availability depends on chord state being initialized — which happens after calibration in `App.tsx` line 65. The chroma guard is functionally correct.

#### Flow 2: Canvas shows family ring colors → instruments clustered by family → edges animate by type

**Status: COMPLETE**

Steps verified:
1. `CanvasRenderer` constructor: `INSTRUMENT_FAMILIES[instrument]` → family → `FAMILY_SORT_ORDER[family]` → ring instruments sorted by family before `computeNodePositions`
2. Render loop line 610–611: `INSTRUMENT_FAMILIES[instrument]` → `FAMILY_RING_COLOR[family]` → `ringColor` passed to `drawNode`
3. `drawNode` receives `ringColor` and draws 3px stroke ring outside fill circle
4. `buildPairs` generates pair tuples → `drawCommunicationEdges` receives `pairs` and `edgeAnimStates`
5. Per animated edge in Pass 3: `e.edgeType` branches to `rhythmic`, `melodic`, or `support` animation — each with distinct visual behavior:
   - `rhythmic`: opacity + lineWidth boosted by `beatPulseIntensity` (from `this.beatPulse / 4`)
   - `melodic`: flowing gradient with animating midpoint stop
   - `support`: slow sine-wave opacity breathing at 0.0025 rad/ms

The `beatPulseIntensity` wiring: `CanvasRenderer.ts` line 578 passes `this.beatPulse / 4` as the last argument to `drawCommunicationEdges`. The function signature at line 134 receives it as `beatPulseIntensity`. This normalizes the `[0,4]` range from beat detection to `[0,1]` for the edge boost calculation.

#### Flow 3: Edge rendering handles all valid pair keys without fallback; invalid keys logged and skipped

**Status: COMPLETE**

All 28 standard pairs for up to 8 instruments have entries in `EDGE_TYPE`. A lineup smaller than 8 only generates the subset of pairs for its instruments, all of which are covered.

## Risk Notes

**Low risk — `DisambiguationState` not re-initialized on lineup change post-calibration**

File: `/Users/seijimatsuda/jazz_learning/src/App.tsx` line 59

`initDisambiguationState()` runs once at calibration. If the user changes the lineup after calibration (e.g., via `BandSetupPanel`), the `CanvasRenderer` is correctly rebuilt (Phase 14 reactive lineup), but `audioStateRef.current.disambiguation` keeps its old ring buffers. The `hasInstrumentPair` guards in `DisambiguationEngine.ts` prevent incorrect disambiguation for absent instruments, and new instruments start accumulating fresh data immediately. No incorrect output occurs, but the first ~2 seconds of data for newly added instruments may see stale tremolo RMS history. Acceptable within current design constraints.

**Low risk — `getInstrumentPairKey` in CanvasRenderer does not handle `trombone` mapped to `horn_section` key**

File: `/Users/seijimatsuda/jazz_learning/src/canvas/CanvasRenderer.ts` lines 79–105

`getInstrumentPairKey` returns `'horn_section'` for `trumpet` but has no case for `trombone` mapping to `horn_section`. Trombone is handled by the `trombone_bass` case (only when bass is present), and otherwise returns `null`. When trombone is in a horn section (trombone + trumpet + saxophone without bass), trombone would not receive the `horn_section` confidence dimming. The `DisambiguationEngine` does correctly compute `horn_section` confidence for all 3+ horn lineups — it is only the UI confidence dimming indicator that would miss trombone in bass-absent horn sections. No incorrect audio analysis; only a minor visual indicator gap.
