## Unsatisfied Requirements

### XCUT-01: iOS Safari Compatible (Partial)

The main file upload flow (FileUpload → handleButtonClick) now creates AudioContext synchronously inside the user gesture handler — this was fixed after Phase 1 verification. However, the `loadExample()` path in App.tsx creates AudioContext inside an async/await chain (after `await fetch()`), which breaks the iOS Safari gesture requirement. On iOS, tapping "Load Example" will silently fail to produce audio.

### USER-04: Pre-loaded Example Tracks (Partial)

Infrastructure is complete: `example-info.json` with 5 expert annotations exists, `loadExample()` fetches and loads it, annotations are pre-populated in Zustand, graceful degradation shows an alert when audio file is missing. However, the actual audio file (`public/examples/example-quartet.mp3`) requires manual placement — no real audio file ships with the project.

## Cross-Phase Integration Issues

### INSTRUMENT_ORDER Hardcoded to 4 (HIGH)

`NodeLayout.ts` defines `INSTRUMENT_ORDER = ['guitar', 'drums', 'keyboard', 'bass']` as a static array. `CanvasRenderer` always calls `computeNodePositions(4)` and creates 4 `NodeAnimState` objects. The `BandSetupPanel` lets users configure 2-3 instrument lineups, and `App.tsx` correctly reads the lineup for `initAnalysisState`. But the canvas ignores the lineup entirely.

**Impact:** Non-standard lineups show ghost nodes with no data. Clicking a ghost node pushes the wrong instrument name to `NodeDetailPanel`, which finds no analysis data and shows empty sparkline/pie.

**Fix path:** `CanvasRenderer` constructor should accept the lineup array from Zustand, filter `INSTRUMENT_ORDER` to match, and call `computeNodePositions(lineup.length)`.

### loadExample iOS AudioContext (BLOCKER on iOS)

`App.tsx` lines 109-147: `loadExample` is async and calls `await fetch()` before reaching `createAudioContext()`. The synchronous gesture link is broken. Desktop browsers don't enforce this restriction, so it works on Chrome/Firefox desktop.

**Fix path:** Pre-create AudioContext synchronously at the start of the `loadExample` click handler, before any `await`, using the same pattern as `FileUpload.handleButtonClick`.

## Tech Debt Summary

### Phase 1
- `CanvasRenderer` calls `getByteFrequencyData` 6x/frame (once per node in loop). All calls overwrite the same pre-allocated array, so nodes 1-5 compute energy from stale data. Visual impact is low because Phase 5 replaced the per-band circles with per-instrument nodes that use `getBandEnergy`, but the redundant FFT reads waste CPU.
- `TransportControls` reads `isCalibrated` from ref at render time — works only because `setCalibrating(false)` triggers a re-render as a side effect.

- `InstrumentRoleOverlay.tsx` is dead code (159 lines). Was created to close Phase 2 display gap, replaced by Phase 5 canvas nodes. Should be deleted.
